# Cultural-Specificity Human Validation App

A small Streamlit app for collecting human labels of whether MMLU-Pro-X
multiple-choice questions are **culturally specific** or **universal**.
Used to validate the LLM-as-a-judge labels in §5.3 of our ACL paper on
multilingual IRT.

## What this app does

We used **Gemini-2.5-Flash** to label MMLU-Pro-X items with
`is_culturally_specific` (Boolean), a `specificity_type`, and a `region`.
This app collects independent human re-labels of a sampled subset of those
items, so we can quantify how well the AI judge matches human judgement.

Annotators log in with a username, are assigned a disjoint batch of 50
items, label each item (culturally specific or universal, plus a
specificity type / region / confidence / optional comment), and can leave
and resume at any time. Labels are stored in a Google Sheet.

## What this measures

The **primary** measure is **agreement between human annotators and the
Gemini-2.5-Flash judge**: the downstream analysis script (separate,
written after collection) computes **Cohen's kappa between each human
annotator's labels and Gemini's labels** on every item that human
annotated.

As a **secondary** check, every annotator also labels a shared **30-item
overlap block**, which supports **one inter-annotator (human-human)
kappa** on those 30 items. The remaining 20 items per batch are disjoint
across annotators, for breadth. So each batch is 30 shared + 20 unique =
50 items; with 20 batches that is 30 + 20x20 = **430 distinct items**
validated.

The shared items are shuffled into random positions within each batch and
are not flagged to annotators; the `is_overlap` column of the `batches`
worksheet identifies them for the analysis.

## Google Sheet structure

One Google Sheet with three worksheets. All three auto-create their header
row on first write — you do **not** need to create them by hand.

| Worksheet     | Columns |
|---------------|---------|
| `batches`     | `batch_id`, `item_id`, `position_in_batch`, `is_overlap` |
| `assignments` | `annotator_id`, `batch_id`, `assigned_at` |
| `annotations` | `annotator_id`, `batch_id`, `item_id`, `is_culturally_specific`, `specificity_type`, `region`, `confidence`, `explanation`, `position_in_batch`, `submitted_at` |

`batches` is populated by `prepare_batches.py` before deployment. The app
writes `assignments` (one row per annotator at login) and `annotations`
(one row per submitted item).

## 1. Create a Google Cloud service account

1. Go to <https://console.cloud.google.com/> and create (or pick) a
   project.
2. Enable the **Google Sheets API** for that project:
   APIs & Services → Library → search "Google Sheets API" → Enable.
3. APIs & Services → Credentials → **Create credentials** → **Service
   account**. Give it a name; no roles are required.
4. Open the new service account → **Keys** tab → **Add key** →
   **Create new key** → **JSON**. A `.json` key file downloads — keep it
   private.
5. Note the service account's email — it looks like
   `something@your-project.iam.gserviceaccount.com`.

## 2. Create and share the Google Sheet

1. Create a new Google Sheet (any name). You do not need to add any tabs.
2. Copy its ID from the URL:
   `https://docs.google.com/spreadsheets/d/`**`<SHEET_ID>`**`/edit`.
3. Click **Share** and share the sheet with the service account email
   from step 1.5, with **Editor** permission.

## 3. Configure secrets locally

```bash
cd cultural_validation_app
cp .streamlit/secrets.toml.example .streamlit/secrets.toml
```

Edit `.streamlit/secrets.toml`:

- Set `GSPREAD_SHEET_ID` to your sheet ID.
- Fill the `[gcp_service_account]` block with the fields from the
  downloaded JSON key file (`type`, `project_id`, `private_key_id`,
  `private_key`, `client_email`, etc.). Copy the values verbatim — keep
  the `\n` escapes inside `private_key`.

`.streamlit/secrets.toml` is git-ignored; never commit it.

## 4. Build the item pool and batches

`prepare_batches.py` builds the joined MCQ pool and uploads the `batches`
worksheet. It needs the `datasets` library (for the one-time HuggingFace
download) in addition to the app dependencies:

```bash
pip install -r requirements.txt datasets
python prepare_batches.py
```

What it does:

1. Reads the four ranker CSVs from `../results/cultural/subsets/`
   (`irt_ratio.csv` = airatio, `acc_max_minus_mean.csv`,
   `acc_gap_en.csv`, `random_items.csv`) and takes the deduplicated union
   of their `item_id`s.
2. Fetches the English MCQ for each item from the HuggingFace dataset
   `li-lab/MMLU-ProX` (`en` / `test` split).
3. Writes `cultural_eval_pool.csv` (`item_id`, `question_text`,
   `option_a`…`option_j`, `correct_answer_letter`, `subject_category`).
4. Samples uniformly at random (`SEED = 42`): a shared
   `OVERLAP_SIZE = 30` item block plus `NUM_BATCHES = 20` disjoint blocks
   of `UNIQUE_PER_BATCH = 20` items. Each annotator's batch is the 30
   shared items + 20 unique items = `BATCH_SIZE = 50`, shuffled together
   and written (with an `is_overlap` flag) to the `batches` worksheet.

Use `python prepare_batches.py --csv-only` to build the CSV and inspect it
before touching the Google Sheet.

**Commit `cultural_eval_pool.csv` to the repo** — the deployed app reads
it at runtime.

## 5. Run locally

```bash
cd cultural_validation_app
streamlit run app.py
```

The app opens at <http://localhost:8501>.

## 6. Deploy to Streamlit Community Cloud

1. Push this repository (including `cultural_eval_pool.csv`, **excluding**
   `.streamlit/secrets.toml`) to GitHub.
2. Go to <https://share.streamlit.io/> and sign in with GitHub.
3. Click **Create app** → **Deploy a public app from GitHub**.
4. Select the repository and branch. Set **Main file path** to
   `cultural_validation_app/app.py`.
5. Open **Advanced settings → Secrets** and paste the entire contents of
   your local `.streamlit/secrets.toml` (the `GSPREAD_SHEET_ID` line and
   the `[gcp_service_account]` block).
6. Click **Deploy**. Streamlit installs `requirements.txt` automatically.
7. Share the resulting `*.streamlit.app` URL with your annotators.

Editing the secrets later: app dashboard → **⋮** → **Settings → Secrets**.

## Monitoring annotations during collection

- **Open the Google Sheet** — the `assignments` and `annotations`
  worksheets update live as people work. `assignments` shows who has a
  batch; `annotations` accumulates one row per submitted item.
- **Per-annotator progress**: in the `annotations` worksheet, count rows
  per `annotator_id`; 50 rows means that annotator finished their batch.
- **Capacity**: 20 batches support up to 20 annotators. When all 20 are
  assigned, a new username sees a "contact the researcher" message — to
  add more capacity, raise `NUM_BATCHES` in `prepare_batches.py` and
  re-run it (this rewrites the `batches` worksheet).
- **Export for analysis**: File → Download → CSV on the `annotations`
  worksheet, or read it with `gspread`. The kappa-vs-Gemini analysis is a
  separate script run after collection. The human-human kappa is computed
  on the 30 items flagged `is_overlap = TRUE` in the `batches` worksheet.

## Notes on behaviour

- **Resume**: a returning annotator who re-enters the same username is
  given back their original batch and continues at the first un-annotated
  item — refreshing the page mid-batch is safe.
- **Username collision**: the same username is treated as the same
  person; their annotations accumulate. Annotators are warned on the
  landing page to pick a unique username.
- **Concurrent annotators**: gspread row appends are safe under
  concurrent writes; no locking is used.
- **Performance**: the static `batches` worksheet and the item pool are
  cached, and per-annotator progress is read from `annotations` once when
  the page loads and then tracked in the session. Clicking buttons (incl.
  the Yes/No toggle) reruns with no Google Sheets round-trip; only login
  and submitting an item touch the sheet.
- This is not real authentication — username-on-session only, which is
  sufficient for a closed annotation study.
